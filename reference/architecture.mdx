## Technology stack

## Application structure

```text
AMH_SISTEMA_OFICIOS/
├── run.py                        # Entrypoint — calls crear_aplicacion()
└── app/
    ├── __init__.py               # Application factory, blueprint registration, extensions
    ├── config.py                 # Config class — reads all settings from .env
    ├── db.py                     # PyMySQL connection pool using Flask's g object
    ├── routes/
    │   ├── auth.py               # bp_auth — /auth prefix (login, logout)
    │   ├── oficios.py            # bp_oficios — /oficios prefix (gestor, subdirector, JUD flows)
    │   └── admin.py              # bp_admin — /admin prefix (admin panel, user management)
    ├── models/
    │   ├── oficio.py             # Raw SQL queries: oficios, documentos, historial, peticiones
    │   ├── usuario.py            # Raw SQL queries: usuarios, roles, auth lookups
    │   └── catalogo.py           # Raw SQL queries: cat_roles, cat_areas, cat_estatus
    ├── services/
    │   ├── oficio_service.py            # ServicioOficio — file storage, transaction orchestration
    │   ├── servicio_autenticacion.py    # ServicioAutenticacion — login validation
    │   └── email_service.py            # Email dispatch functions using Flask-Mail
    ├── utils/
    │   └── pdf_tools.py          # crear_sello_acuse(), estampar_acuse_en_disco()
    ├── templates/
    │   ├── auth/                 # Jinja2 templates for login/logout
    │   ├── oficios/              # Jinja2 templates for oficio flows (gestor, subdirector, JUD)
    │   └── admin/                # Jinja2 templates for admin panel
    └── static/
        ├── css/
        ├── js/
        ├── images/
        └── uploads/              # Uploaded files (oficios and peticiones)
```

<Note>
  The application uses the **factory pattern**. `run.py` calls `crear_aplicacion()` in `app/__init__.py`, which instantiates `Flask(__name__)`, loads configuration from `Config`, registers the database teardown hook, initializes Flask-Login and Flask-Mail, and registers all three blueprints. This pattern makes the application testable and avoids circular imports — routes import from models and services, not from the top-level `app` package directly.
</Note>

## Request flow

```text
Browser
  └─► Flask route (Blueprint handler)
        └─► Service layer  (ServicioOficio, ServicioAutenticacion, email_service)
              └─► Model/DB layer  (raw SQL via PyMySQL cursor)
                    └─► MySQL database
  ◄─── Jinja2 template rendered response
```

The three blueprints divide responsibilities cleanly:

| Blueprint | URL prefix | Role |
|-----------|-----------|------|
| `bp_auth` | `/auth` | Login and logout |
| `bp_oficios` | `/oficios` | All oficio and petición workflows |
| `bp_admin` | `/admin` | Admin panel, user management, oficio deletion |

## Authentication

Flask-Login manages user sessions via a signed cookie. The `login_view` is set to `"auth.login"`, so unauthenticated requests to `@login_required` routes are redirected automatically.

The `user_loader` callback in `app/__init__.py` reloads the user object from the database on every request using the session cookie's user ID:

```python
@login_manager.user_loader
def cargar_usuario(id_usuario):
    from app.models.usuario import buscar_usuario_por_id
    return buscar_usuario_por_id(id_usuario)
```

## Transaction pattern

Services open a shared database connection from Flask's `g` object, call `conexion.begin()` explicitly (PyMySQL connections are created with `autocommit=False`), then pass the cursor to multiple model functions. A single `conexion.commit()` finalizes all writes atomically. Any exception triggers `conexion.rollback()`.

Example — oficio creation writes to three tables in one transaction:

```python
conexion.begin()
with conexion.cursor() as cursor:
    id_oficio = crear_oficio_db(cursor, datos_oficio)        # INSERT INTO oficios
    guardar_documento_db(cursor, id_oficio, ...)              # INSERT INTO documentos_oficio
    registrar_historial_db(cursor, id_oficio, ...)            # INSERT INTO historial_oficios
conexion.commit()
```

If the folio already exists, PyMySQL raises `IntegrityError` (MySQL error 1062), which is caught explicitly to return a user-facing duplicate-folio message rather than a generic error.

## Database connection

`app/db.py` manages connections per-request using Flask's `g` object:

```python
def obtener_conexion():
    if "conexion_mysql" not in g:
        g.conexion_mysql = pymysql.connect(
            host=current_app.config["MYSQL_HOST"],
            user=current_app.config["MYSQL_USER"],
            password=current_app.config["MYSQL_PASSWORD"],
            database=current_app.config["MYSQL_DB"],
            port=current_app.config["MYSQL_PORT"],
            cursorclass=pymysql.cursors.DictCursor,
            autocommit=False,
        )
    return g.conexion_mysql
```

`configurar_base_datos(app)` registers `cerrar_conexion` with `app.teardown_appcontext`, ensuring the connection is closed after every request regardless of outcome.
